## What Went Well

### 1. Speed of delivery was excellent
Eight stories covering anonymous sessions, GDPR cookie consent, Google OAuth, email/password signup, team invite flow, and login/logout — all delivered cleanly. The progressive build approach (each story handing off clean context to the next) kept velocity high and rework minimal.

### 2. Auth callback extensibility decision (Story 1.5)
The auth callback route was designed in Story 1.5 to accommodate all four future branches (anonymous upgrade, team invite, recovery, default) before they were implemented. Stories 1.6, 1.7, and 1.8 plugged in without merge conflicts or rework. This is the pattern to repeat when infrastructure will be extended by multiple future stories.

### 3. Result\<T> discipline held for 8 stories
Every Server Action and DAL function returns `Result<T>` — zero thrown exceptions across async boundaries. This was never enforced by the type system alone; it required consistent attention under delivery pressure.

### 4. Progressive test accumulation
24 → 37 → 43 → 67 → 70+ unit/integration tests across the epic, plus E2E tests on every story with visible UI. Co-located test convention (`filename.test.ts`) made addition natural. E2E tests requiring live DB rows were correctly skipped rather than failing — pragmatic and professional.

### 5. Security-first from day one
Enumeration protection on login, httpOnly cookies for sensitive values (`anon_upgrade_id`), GDPR consent threaded through the OAuth path, server-side ToS validation that cannot be bypassed via direct API calls, generic auth error messages. These are things teams typically bolt on later; they were built correctly from the start.

### 6. Accessibility — 0 violations across all UI stories
`axe-playwright` run consistently on every story with a visible UI. WCAG 2.1 AA compliance confirmed at each step rather than discovered as a late-stage debt item.

## What Could Have Gone Better

### 1. Lint/TypeScript issues caught post-implementation, not during
The git log shows multiple corrective commits (`fix: suppress set-state-in-effect`, `fix: remove unused import`, `fix: restore useEffect import`) indicating ESLint/TS issues were discovered at push time rather than during development. The Husky pre-push hook is the right safety net, but catching these earlier would reduce noise.

**Root cause:** Dev agents completing implementation without running the full lint/TS check locally during the story.

### 2. Dev Agent Records not completed before status update
Story 1.6's Dev Agent Record has no Completion Notes and no File List. The story file itself was left at `ready-for-dev`. The implementation was verified present in the codebase, but the process break means future agents have no documented record of what was changed or why. Story 1.1 and 1.2 story files also remain at `review` status while sprint-status shows `done`.

**Root cause:** Sprint-status updated directly without completing the story file.

### 3. Manual dashboard steps are a production blocker with no central tracking
Story 1.5 contains three unchecked manual tasks (Supabase custom SMTP, redirect allow-list, Google Cloud Console OAuth redirect URI). Without these, Google OAuth and email verification are both non-functional in production. These are buried in a story dev notes section with no central setup guide.

### 4. Third-party API knowledge drift — Zod v4 breaking changes
Story 1.5 hit Zod v4 API changes (`z.email()` vs `z.string().email()`, different error message parameter format) mid-implementation. CLAUDE.md mandates Context7 doc fetching before implementing against any third-party library. This rule was not followed, causing a detour.

### 5. Four pre-existing TypeScript errors untracked
Story 1.4 noted "4 pre-existing TypeScript errors in other files not introduced by this story" and the issue was never followed up. The files affected and whether they impact runtime behaviour is unknown. These were carried into Epic 2.

## Key Insights

1. **Extension-point design beats retrofitting.** The auth callback contract decision in 1.5 saved real effort across 1.6, 1.7, and 1.8. For any infrastructure that will be extended by 2+ future stories, design the extension points explicitly before the first story closes.

2. **Story completion = code + record.** A story is not done if the Dev Agent Record is empty. The record is the handoff document for the next story's agent. An empty record is the same failure mode as incomplete code — future stories lose context.

3. **Manual steps need a home outside story files.** Story dev notes are ephemeral reference during implementation. Manual one-time setup steps (dashboard config, cloud console URIs, environment variables) need a persistent, findable location — a setup checklist — or they fall through the gap.

4. **Use Context7 for third-party libraries without exception.** Training-data knowledge of library APIs is unreliable across major versions. The Zod v4 detour was avoidable.
