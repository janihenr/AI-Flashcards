## What Went Well

### 1. GDPR compliance delivered correctly, not as checkbox compliance

Data export with async Edge Function, atomic account deletion (Drizzle transaction covering profile/decks/notes/reviews), session invalidation via `adminClient.auth.admin.signOut(userId, { scope: 'global' })`, and deleted-account re-login blocked at both OAuth callback and email/password entry points. This is a complete, correct implementation — not scaffolding.

### 2. Epic 1 process fixes held

P1 (lint/TS gate) and P2 (complete Dev Agent Records) were both sustained across all 6 stories. No story shipped with an empty record. No corrective lint commits in the git log.

### 3. Pattern library grew and held consistency

`useActionState` vs `useTransition` (form state vs stateless optimistic), WCAG `<section aria-labelledby>` landmarks, `Result<T>` across all Server Actions and DAL functions, Context7 usage for third-party APIs — all applied consistently across 6 stories with no drift.

### 4. First Edge Function delivered

`process-data-export` establishes the pattern for async background work in the project: `after()` from `next/server` triggers the Edge Function post-response; job status tracked in `data_export_jobs`; failure leaves job `pending` for user-initiated retry. This pattern is reusable.

### 5. Cross-story design awareness was good

Stories 2.2 through 2.6 each explicitly noted what the next story would add to the same page, leaving clean extension points rather than placeholders that required restructuring.

## What Could Have Gone Better

### 1. Deferred E2E testing is a workflow design flaw

Stories 2.2, 2.3, 2.4, 2.5, and 2.6 all contain deferred manual verification items blocked because the app had no navigable UI (no `/decks` page, login stuck on email auth, Google OAuth redirects to `/terms`). These were marked `done` in sprint-status with the validation loop still open.

**Root cause:** The workflow has no defined phase for "run deferred acceptance tests when the app is navigable." This is a structural gap, not a discipline issue.

### 2. `notes`-in-`decks.ts` caused friction across three consecutive stories

Stories 2.4, 2.5, and 2.6 each contain a CRITICAL warning that `notes` is defined in `decks.ts`, not a separate `notes.ts` file. Every other table has its own file (`users.ts`, `cards.ts`, `reviews.ts`, `teams.ts`, `exportJobs.ts`). The naming anomaly either caused real import errors or near-misses in multiple stories.

**Root cause:** Schema file naming convention not followed when `notes` was added to `decks.ts`. Fix is structural and low-risk.

### 3. Technical debt T1 carried through the entire epic

Four pre-existing TypeScript errors from Epic 1 (Story 1.4) were noted as "inherited" in Epic 2 stories without being addressed. They completed two full epics unresolved.

---

## Key Insights

### 1. "Done" requires a closed validation loop

If a story's acceptance criteria require a running UI that doesn't exist yet, the workflow must explicitly account for a deferred-validation phase. Marking `done` without that phase closed creates invisible debt. Going forward: deferred E2E tests should be tracked as explicit tasks, not buried in completion notes.

### 2. Schema file naming is load-bearing

A single file naming anomaly (`notes` in `decks.ts`) cascaded into three CRITICAL warnings and repeated friction. In a DAL-heavy codebase where agents reference file paths constantly, the convention "one table = one file" needs to be a hard rule.

### 3. Carried technical debt doesn't disappear — it compounds

T1 (4 TS errors) was noted in the Epic 1 retro, carried through all of Epic 2, and will continue into Epic 3 unless explicitly addressed before starting. Each story that inherits it without resolving it makes resolution slightly harder.
